  对话回顾

  背景

  你在 RK3588 上部署了 Qwen3-VL 4B（NPU 推理），并搭建了 TTS
  语音播报桥接系统，目标是实现低延迟的语音助手交互。

  时间线

  1. MeloTTS 阶段 — Qwen + MeloTTS 语音桥接，解决编码/JSON/依赖问题，优化到 ~1s
  合成。但 MeloTTS-RKNN2 decoder 与 Qwen3-VL 抢 NPU，并行时冲突。
  2. Sherpa-onnx 阶段 — 换 sherpa-onnx（纯 CPU），端口 8011，实现边生成边播 +
  profiling + 录音时间轴。
  3. ZipVoice 阶段 — 追求多音色/语音克隆，换 ZipVoice ONNX，端口 8012。但 RTF
  1.4-5.2，长句 23s+，太慢。
  4. Piper 阶段（当前） — 换 Piper ONNX (zh_CN-xiao_ya-medium)，RTF 0.68，纯
  CPU。花大量精力解决中文依赖（g2pw/torch/pypinyin shim）。

  最终实现

  - 交互脚本：/home/linaro/Qwen/scripts/23_interactive_piper_persistent_qwen.py
    - Qwen 常驻 + Piper 8011 端口
    - 两级流水线：synth_worker（合成队列）→ play_worker（播放队列）
    - 播放第 1 句时后台合成第 2 句

  最后解决的问题：首句延迟

  问题：首句合成+播报需要 ~9s，其中 Piper subprocess 冷启动占 ~2.5s。

  根因：piper_tts_engine.py 每次 /synthesize 都用 subprocess.run(piper 
  CLI)，每次重新启动进程、import torch/onnxruntime/g2pw、加载 63MB ONNX 模型。

     背景

     当前 8011 TTS 服务使用 Piper 后端（TTS_ENGINE=piper），用户想切换到
     Sherpa-onnx VITS 后端以获得更好的中文语音质量。Sherpa-onnx VITS
     模型已就绪（/home/linaro/models/vits-melo-tts-zh_en/，162MB ONNX + lexicon
     + tokens + FST 规则）。

     关键发现：零代码改动

     两个后端通过同一 FastAPI 服务 (sherpa_tts_server.py) 暴露完全相同的 API：
     - GET /health — 返回后端信息
     - POST /synthesize — {"text":..., "output":..., "sid":..., "speed":...} →
     返回 WAV 元数据
     - POST /speak — 合成 + 可选播放

     23_interactive_piper_persistent_qwen.py 只调用 /synthesize 端点，不依赖任何
      Piper 特定逻辑。无需修改流水线代码。
